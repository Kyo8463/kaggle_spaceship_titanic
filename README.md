## 結果

| 指標 | スコア |
|------|--------|
| Kaggle Public Score | **0.79939** |
| CV Accuracy（5-fold） | 0.8060 ± 0.0075 |

## 問題の概要

宇宙船タイタニック号が異次元と衝突し、乗客が別の次元に転送されたかどうかを予測する2値分類タスク。

- **訓練データ**: 8,693件
- **テストデータ**: 4,277件
- **目的変数**: `Transported`（True / False、約50/50の均衡クラス）

---

## 分析の流れ

```
データ読み込み → EDA → 仮説検証 → 特徴量エンジニアリング → ベースラインモデル → ハイパーパラメータ調整 → 提出
```

## 1. 探索的データ分析（EDA）

### 欠損値

全特徴量の欠損率は約2%と低く、LightGBM の欠損値自動処理に委ねる方針とした。

### 目的変数の分布

True と False がほぼ50/50であり、クラス不均衡への対処は不要と判断した。

### 数値変数の分布

`Transported` で色分けしたヒストグラムから2点を読み取った。

- **支出カラム（RoomService, Spa, VRDeck）**: 分布が右裾に大きく歪んでおり、大半の乗客が0円。これらに支出した乗客ほど転送されていない傾向が強い。
- **Age**: 若年層ほど転送率が高い。また、`Age=0` が178件存在しており、実際の0歳ではなく「年齢不明」を意味するデータ入力エラーである可能性が高かった。

### カテゴリ変数の傾向

| 特徴量 | 読み取り内容 |
|--------|------------|
| `CryoSleep` | 最も強い予測変数。True の場合、約82%が転送されている |
| `HomePlanet` | Europa 出身者の転送率が Earth・Mars より顕著に高い |
| `Destination` | 55 Cancri e だけが他の2目的地と異なる傾向を示す |
| `VIP` | VIP 乗客の転送率はやや低いが、サンプル数が199件と少なく統計的信頼性に注意が必要 |

### 相関分析

支出カラムと `Transported` の相関係数を個別に計算したところ、方向性の違いが明確に現れた。

| カラムグループ | Transported との相関 |
|--------------|-------------------|
| RoomService, Spa, VRDeck | **-0.207 〜 -0.245**（強い負の相関）|
| FoodCourt, ShoppingMall | **+0.046**（弱い正の相関）|

→ すべてを合算した `TotalSpend` を使うと、この方向性の違いが打ち消し合う。

### Deck の分析と交絡変数

Deck B・C の転送率が高く見えたが、`CryoSleep` を制御して再分析した結果、**交絡変数**の存在が判明した。

- Deck B・C には Europa 出身者がほぼ100%居住
- Europa 出身者は CryoSleep 率が最も高い
- 転送率を高めていたのは Deck そのものではなく、CryoSleep だった

> **学び**: ある変数の効果を解釈するとき、他の変数を制御して再確認することが重要。

## 2. 仮説検証

特徴量の採否はすべて5-fold CV による精度比較で判断した。

### 仮説1 — Age=0 の扱い方

| アプローチ | CV Accuracy |
|-----------|-------------|
| A: 何もしない（ベースライン） | 0.7957 |
| B: Age=0 を NaN に置換 | **0.7967（+0.001）** |
| C: Age_is_zero フラグを追加 | 0.7958 |
| D: B + C（両方） | 0.7963 |

**採用**: Age=0 を NaN に置換。178件の0歳は記録エラーと判断し、欠損値として扱うことで精度が向上した。

### 仮説2 — 支出カラムの集約方法

| アプローチ | CV Accuracy |
|-----------|-------------|
| A: TotalSpend（5列の合計） | 0.7957 |
| B: Spend_RSV + Spend_FC（2グループ） | **0.7997（+0.004）** |
| C: 5カラム個別 | 0.7993 |

**採用**: 2グループに分割。相関の方向が逆のカラムを合算すると情報が損なわれることを精度比較で確認した。

### 仮説3 — Destination: 3カテゴリ vs. 二値フラグ

| アプローチ | 精度変化 |
|-----------|---------|
| 3カテゴリのまま使用 | -0.0008 |
| `Is55Cancri` 二値フラグ | **+0.0009** |

**採用**: 55 Cancri e のみ転送率が異なり、残り2目的地は同様の傾向。二値フラグの方がモデルに伝わりやすかった。

### 仮説4 — GroupSize と IsAlone の組み合わせ

| アプローチ | 精度変化 |
|-----------|---------|
| GroupSize のみ | +0.0011 |
| IsAlone のみ | +0.0006 |
| GroupSize + IsAlone 両方 | **+0.0017** |

**採用**: 両方。グループの人数と「一人旅かどうか」は補完的な情報を持っていた。

### 仮説5 — Cabin の部屋番号

数値の部屋番号を追加した結果、精度が **-0.03** 以上低下した。値の範囲が広く、転送率との相関がほぼゼロのノイズ特徴量だった。

**不採用**: 削除。

### 仮説6 — 交互作用特徴量（HomePlanet × CryoSleep）

`Europa × CryoSleep` や `Mars × CryoSleep × DeckF` などの交互作用特徴量を手動で作成したが、精度向上はなかった。

**理由**: LightGBM はツリーの分岐を通じて交互作用を自動的に学習するため、手動エンコードは情報を追加しない。

### 仮説7 — 苗字からの FamilySize 推定

`Name` カラムから苗字を抽出し、同一苗字のグループサイズを計算したが、PassengerId ベースの GroupSize との一致率は9.9%に留まり、転送率との関係も弱かった。

**不採用**: 削除。

## 3. 最終特徴量セット

```python
FEATURES = [
    'HomePlanet',   # Europa / Earth / Mars
    'CryoSleep',    # 最強の予測変数
    'Age',          # Age=0 を NaN に置換済み
    'VIP',
    'Deck',         # Cabin を '/' で分割して取得
    'Side',         # Cabin を '/' で分割して取得（P / S）
    'GroupSize',    # PassengerId の先頭グループ番号から算出
    'IsAlone',      # GroupSize == 1 の二値フラグ
    'HasSpend',     # 支出が1円でもあれば 1
    'Spend_RSV',    # RoomService + Spa + VRDeck（負の相関グループ）
    'Spend_FC',     # FoodCourt + ShoppingMall（正の相関グループ）
    'Is55Cancri',   # Destination == '55 Cancri e' の二値フラグ
]
```

## 4. モデル

### アルゴリズム: LightGBM

以下の理由で選択した。

- 欠損値をそのまま扱える（補完不要）
- カテゴリ変数のネイティブサポート
- 特徴量の交互作用を自動学習
- 勾配ブースティングの中でも学習が高速

### ベースライン（デフォルトパラメータ）

```
CV Accuracy: 0.7974 ± 0.0054
```

### Optuna によるハイパーパラメータ調整

**TPE（Tree-structured Parzen Estimator）** サンプラーで100回試行。各試行で5-fold CV を実行し、平均 AUC を最大化。過学習防止のため、各 fold に `early_stopping（stopping_rounds=50）` を適用。

| パラメータ | 探索範囲 | 採用値 | 探索範囲の理由 |
|-----------|---------|-------|--------------|
| `learning_rate` | 0.01〜0.30（対数） | 0.0136 | 小さいほど丁寧に学習。対数スケールで幅広く探索 |
| `num_leaves` | 20〜300 | 195 | モデルの複雑さを制御。デフォルト31を中心に広めに |
| `max_depth` | 3〜12 | 6 | 木の深さ制限。深すぎると過学習 |
| `min_data_in_leaf` | 10〜100 | 28 | 1葉の最小サンプル数。大きいほど汎化しやすい |
| `feature_fraction` | 0.5〜1.0 | 0.744 | 1ツリーあたりの特徴量割合。ランダム性で過学習抑制 |
| `bagging_fraction` | 0.5〜1.0 | 0.749 | 1ツリーあたりのデータ割合。ランダム性で過学習抑制 |
| `lambda_l1` | 0〜10 | 0.313 | L1正則化。不要な特徴量の重みを0に近づける |
| `lambda_l2` | 0〜10 | 1.402 | L2正則化。重みを全体的に小さく抑える |

### 最終 CV 結果

| Fold | AUC | Accuracy | 最適イテレーション数 |
|------|-----|----------|-----------------|
| 1 | 0.8956 | 0.8131 | 455 |
| 2 | 0.8865 | 0.8016 | 539 |
| 3 | 0.8948 | 0.8114 | 547 |
| 4 | 0.9017 | 0.8107 | 583 |
| 5 | 0.8816 | 0.7934 | 373 |
| **平均** | **0.8920 ± 0.0071** | **0.8060 ± 0.0075** | — |

最適イテレーション数が全 fold で373〜583（上限1000を大きく下回る）であり、early stopping が正常に機能していることを確認。

### 汎化性能の確認

| CV Accuracy | Kaggle Public Score | 差 |
|-------------|---------------------|----|
| 0.8060 | **0.79939** | 0.0066 |

差が0.01以内であり、過学習なく未知データに汎化していることを確認できた。

## 5. 学んだこと

1. **相関の方向性を確認する**: 支出カラムは正・負の相関が混在しており、合算より2グループに分けることで精度が向上した。合算するだけでは情報が失われる。

2. **交絡変数に注意する**: Deck B/C の高転送率は、CryoSleep という交絡変数によるものだった。表面的な傾向だけでなく、他の変数を制御して再分析することが重要。

3. **木モデルは交互作用を自動学習する**: 手動で作成した交互作用特徴量はすべて精度向上に貢献しなかった。LightGBM にとっては冗長な情報だった。

4. **カテゴリのまとめ方は精度に影響する**: 3カテゴリの Destination では精度が下がったが、特異な1カテゴリだけの二値フラグにすると改善した。カテゴリのグループ化方法が重要。

5. **特徴量は検証してから採用する**: すべての特徴量を CV で検証してから採用した。この習慣により、ノイズ特徴量（部屋番号、FamilySize）の混入を防ぐことができた。
